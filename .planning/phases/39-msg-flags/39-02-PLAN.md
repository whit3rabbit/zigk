<objective>
Implement MSG_WAITALL for TCP sockets and add integration tests for all three MSG flags (MSG_PEEK, MSG_DONTWAIT, MSG_WAITALL).

Purpose: MSG_WAITALL is used by protocol implementations that need exact-length reads (e.g., reading a 4-byte length header followed by exactly N bytes of payload). Without it, TCP recv may return partial data and the application must manually loop -- which is error-prone and defeats the purpose of the kernel providing this semantic. Tests validate all three flags from Phase 39.

Output: tcpRecvWaitall accumulation function, MSG_WAITALL dispatch in sys_recvfrom, and comprehensive tests for all MSG flag behaviors.
</objective>

**1. Add `tcpRecvWaitall` in `tcp_api.zig`**:

This function loops calling `tcp.recv` to accumulate exactly `buf.len` bytes. It must handle:

- **Accumulation:** Maintain `offset: usize = 0`. Each successful `tcp.recv(tcb, buf[offset..])` returns `n` bytes; add `n` to `offset`. Loop until `offset == buf.len`.
- **EOF:** If `tcp.recv` returns 0 (FIN received), break and return `offset` (partial data or 0 if nothing received).
- **WouldBlock (buffer empty):** Block via scheduler. Use the same pattern as `udp_api.recvfromIp` blocking loop:
  1. Get `scheduler.blockFn` and `scheduler.currentThreadFn`
  2. Disable interrupts, acquire `sock.lock`, set `sock.blocked_thread`, try `tcp.recv`, if WouldBlock release lock, call `block_fn()`, retry
  3. After block_fn returns, clear `blocked_thread`, restore interrupts
- **SO_RCVTIMEO:** Capture `start_tsc = clock.rdtsc()` before the loop. Convert `sock.rcv_timeout_ms` to microseconds. Check `clock.hasTimedOut(start_tsc, timeout_us)` at each iteration. On timeout: if `offset > 0` return `offset` (partial); if `offset == 0` return `SocketError.TimedOut`.
- **EINTR (signal check):** After each block/wakeup, check `hasPendingSignal()` (import from poll module or net.zig). If signal pending: if `offset > 0` return `offset` (partial); if `offset == 0` return `SocketError.WouldBlock` (the syscall layer maps this to EINTR when appropriate -- or use a dedicated signal error if available).

Signature:
```zig
pub fn tcpRecvWaitall(sock_fd: usize, buf: []u8) errors.SocketError!usize
```

The function acquires the socket via `state.acquireSocket`, gets the `tcb` from `sock.tcb`, and loops. It needs access to `sock.rcv_timeout_ms` and `sock.blocked_thread`.

**IMPORTANT:** The `tcp.recv` function acquires `state.lock` then `tcb.mutex` internally. The WAITALL loop must NOT hold `state.lock` or `tcb.mutex` across iterations -- each `tcp.recv` call manages its own locking. The blocking pattern (disable IRQs, acquire `sock.lock`, set `blocked_thread`) uses `sock.lock` which is at level 6 -- this is fine because `tcp.recv` acquires `tcp_state.lock` (5) then `tcb.mutex` (7), and we release `sock.lock` before calling `tcp.recv`.

Actually -- re-read the UDP blocking pattern more carefully. The UDP pattern acquires `sock.lock`, tries dequeue, releases lock, then blocks. For TCP WAITALL, the pattern should be:
1. Try `tcpRecv(sock_fd, buf[offset..])` (this handles its own locking internally)
2. If success, add to offset, continue
3. If WouldBlock: enter blocking sub-loop:
   a. Disable interrupts
   b. Acquire `sock.lock`, set `blocked_thread = current_thread`
   c. Release `sock.lock`
   d. Call `block_fn()` (suspends until wakeup from TCP data arrival)
   e. Clear `blocked_thread`, restore interrupts
   f. Check timeout/signal, retry from step 1

This is cleaner because `tcpRecv` manages its own lock acquisition.

**2. Wire MSG_WAITALL in `sys_recvfrom` (net.zig)**:

In the SOCK_STREAM branch (added in 39-01), add another check:
```zig
if ((recv_flags & socket.MSG_WAITALL) != 0) {
    // MSG_WAITALL: accumulate until buf full, EOF, or error
    const received = socket.tcpRecvWaitall(ctx.socket_idx, kbuf) catch |err| {
        return socketErrorToSyscallError(err);
    };
    // copy to user, handle src_addr, return received
}
```

**For SOCK_DGRAM:** MSG_WAITALL is ignored per POSIX. The existing UDP path runs as-is when MSG_WAITALL is set. Add a comment: `// MSG_WAITALL ignored for SOCK_DGRAM per POSIX`.

**Interaction with MSG_PEEK:** If both `MSG_PEEK | MSG_WAITALL` are set, POSIX says MSG_WAITALL is ignored because MSG_PEEK returns without consuming. Dispatch to the MSG_PEEK path. Check MSG_PEEK first: `if (recv_flags & MSG_PEEK != 0) { ... } else if (recv_flags & MSG_WAITALL != 0) { ... } else { ... }`.

**Interaction with MSG_DONTWAIT:** If both `MSG_DONTWAIT | MSG_WAITALL` are set, MSG_DONTWAIT wins. The non-blocking semantics prevent WAITALL from blocking. Dispatch to the MSG_DONTWAIT path. The priority is: MSG_PEEK > MSG_DONTWAIT > MSG_WAITALL > default.

**Test 1: testMsgPeekUdp** -- MSG_PEEK on UDP
1. Create two UDP sockets, bind receiver to loopback:9200
2. Send "hello" from sender
3. Call `recvfromFlags(recv_fd, buf, MSG_PEEK, null)` -- expect 5 bytes, buf == "hello"
4. Call `recvfromFlags(recv_fd, buf, 0, null)` (no flags) -- expect 5 bytes, buf == "hello" (same data, now consumed)
5. Call `recvfromFlags(recv_fd, buf, MSG_PEEK | MSG_DONTWAIT, null)` -- expect EAGAIN (WouldBlock) because queue is now empty

**Test 2: testMsgPeekTcp** -- MSG_PEEK on TCP
1. Create TCP server socket, bind to loopback:9201, listen
2. Create TCP client, connect to server
3. Accept connection on server
4. Client sends "world" via write()
5. Server calls `recvfromFlags(accepted_fd, buf, MSG_PEEK, null)` -- expect up to 5 bytes
6. Server calls `recvfromFlags(accepted_fd, buf, 0, null)` -- expect same data (now consumed)
7. Verify both reads returned the same content

**Test 3: testMsgDontwaitEagain** -- MSG_DONTWAIT returns EAGAIN
1. Create TCP server + client pair (same setup as test 2)
2. Do NOT send any data
3. Client calls `recvfromFlags(client_fd, buf, MSG_DONTWAIT, null)` -- expect error.WouldBlock (errno 11 = EAGAIN)
4. This proves MSG_DONTWAIT overrides the default blocking behavior

**Test 4: testMsgWaitallTcp** -- MSG_WAITALL accumulates full length
1. Create TCP server + client pair
2. Client sends 4 bytes "ABCD"
3. Server calls `recvfromFlags(accepted_fd, buf[0..4], MSG_WAITALL, null)` -- expect exactly 4 bytes returned
4. Verify buf[0..4] == "ABCD"
Note: In the loopback path, data likely arrives in one chunk. This test primarily validates that MSG_WAITALL does not break normal operation. A more rigorous test would send data in multiple small writes with yields between them, but loopback timing makes that unreliable.

**Test 5: testMsgWaitallIgnoredUdp** -- MSG_WAITALL ignored for UDP
1. Create UDP sender/receiver pair
2. Send "hi" (2 bytes)
3. Call `recvfromFlags(recv_fd, buf[0..100], MSG_WAITALL, null)` -- expect 2 bytes returned (not blocking for 100 bytes)
