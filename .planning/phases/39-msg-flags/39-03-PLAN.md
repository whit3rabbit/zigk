**1. scheduler.zig -- Add hasPendingSignal callback**

The transport layer (`tcp_api.zig`) cannot import `sched` or `Thread` directly. It uses the `scheduler.zig` shim which stores function pointers as `?*anyopaque`-based callbacks. Add a 4th callback for signal checking.

Add a new callback type and storage:
```zig
pub const HasPendingSignalFn = *const fn () bool;
var has_pending_signal_fn: ?HasPendingSignalFn = null;
```

Extend `setSchedulerFunctions` to accept and store the 4th parameter:
```zig
pub fn setSchedulerFunctions(wake: WakeFn, block: BlockFn, getCurrent: GetCurrentThreadFn, hasPending: HasPendingSignalFn) void {
    const held = lock.acquire();
    defer held.release();
    wake_thread_fn = wake;
    block_thread_fn = block;
    get_current_thread_fn = getCurrent;
    has_pending_signal_fn = hasPending;
}
```

Add public accessor:
```zig
/// Check if the current thread has pending unmasked signals.
/// Returns false if no callback is registered (safe default: do not interrupt).
pub fn hasPendingSignal() bool {
    const held = lock.acquire();
    defer held.release();
    if (has_pending_signal_fn) |f| return f();
    return false;
}
```

**2. net.zig -- Register the hasPendingSignal callback and add EINTR checks**

```zig
socket.setSchedulerFunctions(wakeBlockedThread, blockCurrentThread, getCurrentThread, hasPendingSignalImpl);
```

Add the implementation function (alongside the other three callback functions around lines 32-48):

```zig
/// Check pending signals on current thread -- callback for socket scheduler shim
fn hasPendingSignalImpl() bool {
```

In `sys_recvfrom`, add EINTR check in the **MSG_PEEK blocking loop** (around line 618-620, after `sched.block()`):
After `sched.block();` and before `sock_for_type.blocked_thread = null;`, add:

```zig
sched.block();
// Check for pending signal after wakeup

    sock_for_type.blocked_thread = null;

continue;
```

Note: the existing `sock_for_type.blocked_thread = null;` on line 619 must come AFTER the signal check but BEFORE the `return error.EINTR`, because we need to clean up the blocked_thread pointer. Move the null assignment inside the signal check branch AND keep one on the continue path, OR restructure using `defer`. The simplest correct pattern is to place `sock_for_type.blocked_thread = null;` immediately after `sched.block()` (before the signal check), which is safe because we are already awake:

```zig
sched.block();
sock_for_type.blocked_thread = null;
if (hasPendingSignal()) {
    return error.EINTR;
}
continue;
```

**3. tcp_api.zig -- Add signal check in tcpRecvWaitall blocking sub-loop**

In `tcpRecvWaitall`, in the scheduler-based blocking path (around lines 546-561), after `block_fn()` returns and `sock.blocked_thread = null`, add a signal check before the timeout check:

Current code (lines 552-561):
```zig
block_fn();
sock.blocked_thread = null;
platform.cpu.restoreInterrupts(irq_state);

// After wakeup: check timeout and continue to retry receive.
if (timeout_us > 0 and clock.hasTimedOut(start_tsc, timeout_us)) {
    if (offset > 0) return offset;
    return errors.SocketError.TimedOut;
}
continue;
```

Change to:
```zig
block_fn();
sock.blocked_thread = null;
platform.cpu.restoreInterrupts(irq_state);

// Check for pending signal after wakeup.
if (scheduler.hasPendingSignal()) {
    if (offset > 0) return offset;
    return errors.SocketError.WouldBlock;
}

// After wakeup: check timeout and continue to retry receive.
if (timeout_us > 0 and clock.hasTimedOut(start_tsc, timeout_us)) {
    if (offset > 0) return offset;
    return errors.SocketError.TimedOut;
}
continue;
```

The return of `WouldBlock` with zero offset is correct here because the syscall layer (`sys_recvfrom`) will then check `hasPendingSignal()` and convert to EINTR. If a signal is pending but some bytes were accumulated, the function returns the partial count (not an error), matching POSIX MSG_WAITALL semantics.
