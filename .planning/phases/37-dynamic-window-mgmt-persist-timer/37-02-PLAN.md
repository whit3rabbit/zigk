<objective>
Add sender SWS avoidance in transmitPendingData() and post-drain window update ACK in api.recv().

Purpose: RFC 1122 S4.2.3.4 requires the sender to hold back tiny segments that are less than MSS and less than half the peer window, unless they exhaust the send buffer. RFC 1122 S4.2.3.3 requires the receiver to proactively send a window update ACK when enough buffer space frees up after a recv() call. Together these prevent silly-window syndrome from both sides.

Output: Updated data.zig with sender SWS gate, updated api.zig with window-update ACK logic.
</objective>

In transmitPendingData(), after the existing Nagle check (the `if (!tcb.nodelay and flight_size > 0 and send_len < effective_mss)` block, which should be around line 85 after Plan 01 changes), add the sender SWS avoidance gate:

```zig
// RFC 1122 S4.2.3.4 Sender SWS avoidance:
// Do not send a segment unless it is a full MSS, covers at least half the
// peer's advertised window, or exhausts all remaining data in the send buffer.
// This is additive to Nagle -- Nagle gates on flight_size, SWS gates on segment size.
const is_full_segment = send_len >= @as(usize, effective_mss);
const half_wnd: usize = if (tcb.snd_wnd > 1) @as(usize, tcb.snd_wnd / 2) else 1;
const is_half_window = send_len >= half_wnd;
const is_last_data = send_len == buffered;
if (!is_full_segment and !is_half_window and !is_last_data) {
    return true; // Hold back: sender SWS avoidance
}
```

Place this AFTER the Nagle check and BEFORE the `if (send_len == 0) return true;` line. The ordering is:
1. Nagle check (existing)
2. Sender SWS check (new)
3. send_len == 0 guard (existing)
4. Build data_buf and send (existing)

Do NOT modify or remove the Nagle check. The two guards are complementary:
- Nagle: suppresses small sends when unacked data is in flight
- SWS: suppresses tiny-window sends even when no data is in flight

<task type="auto">
  <name>Task 2: Add post-drain window update ACK in recv()</name>
  <files>src/net/transport/tcp/api.zig</files>
  <action>
In api.recv(), modify the `if (available > 0)` block (currently lines 207-216) to snapshot the buffer occupancy before draining and send a window update ACK if enough space freed up.

IMPORTANT: Use unscaled byte comparison (recvBufferAvailable) rather than the scaled currentRecvWindow() return value. This avoids window-scaling arithmetic complexity per the research recommendation in Open Question 2. The threshold is MSS bytes of freed space.

Replace the `if (available > 0)` block with:

```zig
if (available > 0) {
    // WIN-03: Snapshot buffer occupancy before drain (unscaled bytes)
    const old_used = tcb.recvBufferAvailable();

    const copy_len = @min(buf.len, available);

    for (0..copy_len) |i| {
        buf[i] = tcb.recv_buf[tcb.recv_tail];
        tcb.recv_tail = (tcb.recv_tail + 1) % c.BUFFER_SIZE;
    }

    // WIN-03: Send window update ACK if freed space grew by >= MSS.
    // Compare in unscaled bytes to avoid window-scaling arithmetic.
    // currentRecvWindow() (called by sendAck -> sendSegment) already
    // includes the WIN-04 SWS floor, so the ACK will carry the correct
    // (possibly zero) window value.
    const new_used = tcb.recvBufferAvailable();
    const freed: usize = if (old_used > new_used) old_used - new_used else 0;
    if (freed >= @as(usize, tcb.mss)) {
        _ = tx.sendAck(tcb);
    }

    return copy_len;
}
```

This runs within the tcb.mutex held scope (acquired at line 198 via state.lock pattern, deferred release at line 200). The sendAck call is safe under this lock -- research confirms this is the correct pattern (Pitfall 3 in RESEARCH.md).

Note: `freed` equals `copy_len` in the normal case (old_used - new_used == bytes drained). The defensive `if (old_used > new_used)` guards against theoretical edge cases with concurrent recv_head writes from the RX path.
