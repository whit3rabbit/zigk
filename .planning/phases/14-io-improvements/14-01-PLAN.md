<objective>
Optimize sendfile to use a larger kernel transfer buffer and eliminate per-chunk heap allocation.

Purpose: The current sys_sendfile allocates and frees a 4KB heap buffer on every call and uses it for read-write loops. This is inefficient for large file transfers. The improvement replaces the per-call heap allocation with a larger (64KB) buffer, matching the chunk size already used by sys_read/sys_write. True zero-copy (direct page mapping between FDs) is not feasible without a page cache layer, but eliminating the heap alloc/free per-call and using a larger transfer window provides a measurable improvement.

Output: Optimized sendfile implementation with a large-transfer integration test.
</objective>

1. **Increase transfer buffer size**: Change `kbuf_size` from 4096 to 65536 (64KB), matching the chunk size used by sys_read and sys_write. This reduces the number of read/write syscall cycles by 16x for large transfers.

2. **Allocate buffer once and reuse**: The current implementation already allocates once per sendfile call -- no change needed here. But rename the constant for clarity:
   - `const kbuf_size = 4096;` -> `const sendfile_buf_size: usize = 64 * 1024;`
   - Update the `heap.allocator().alloc(u8, sendfile_buf_size)` call accordingly.

3. **Maintain correctness**: All existing logic (offset_ptr handling, lock acquisition per chunk, position restoration, short read/write handling, overflow checks) must be preserved exactly as-is. Only the buffer size constant changes.

NOTE: True zero-copy (direct page mapping from source to destination FD) requires a page cache abstraction that this kernel does not have. The VFS operates through FileOps.read/write which copy data into byte slices. Adding a page-level splice path would require a new FileOps method (e.g., splice_pages) and filesystem-level support, which is out of scope for this plan. The 64KB buffer approach is the practical improvement given the current architecture.

```zig
// Test 13: sendfile large transfer - verify multi-page transfer works
pub fn testSendfileLargeTransfer() !void {
    // Open a large source file (shell.elf is >4KB)
    const fd = try syscall.open("/shell.elf", syscall.O_RDONLY, 0);
    defer syscall.close(fd) catch {};

    // Create a pipe for destination
    var pipefd: [2]i32 = undefined;
    try syscall.pipe(&pipefd);
    defer {
        syscall.close(pipefd[0]) catch {};
        syscall.close(pipefd[1]) catch {};
    }

    // Transfer 8KB (larger than old 4KB buffer, exercises multi-chunk path)
    const transfer_size: usize = 8192;
    var offset: u64 = 0;
    const sent = try syscall.sendfile(pipefd[1], fd, &offset, transfer_size);

    // Should have sent the requested amount (shell.elf is large enough)
    if (sent != transfer_size) return error.TestFailed;

    // Verify offset was updated
    if (offset != transfer_size) return error.TestFailed;

    // Read back from pipe and verify first bytes match ELF magic
    var verify_buf: [4]u8 = undefined;
    const read_bytes = try syscall.read(pipefd[0], &verify_buf, verify_buf.len);
    if (read_bytes != 4) return error.TestFailed;
    if (verify_buf[0] != 0x7F or verify_buf[1] != 'E' or verify_buf[2] != 'L' or verify_buf[3] != 'F') {
        return error.TestFailed;
    }
}
```
