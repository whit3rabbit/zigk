<objective>
Enable AT_SYMLINK_NOFOLLOW support in utimensat syscall.

Purpose: The utimensat syscall currently returns ENOSYS when the AT_SYMLINK_NOFOLLOW flag is passed. This prevents programs that want to modify symlink timestamps (common in backup/archive utilities like tar, rsync). The fix is straightforward: the current VFS does not follow symlinks during path resolution (it operates on literal paths), so AT_SYMLINK_NOFOLLOW behavior is already the default. We just need to remove the ENOSYS return and let the call proceed normally. The flag is accepted and the path operates on whatever the literal path refers to (which, for a symlink path, is the symlink entry itself in SFS).

Output: Working AT_SYMLINK_NOFOLLOW in utimensat, updated test.
</objective>

That is: remove the `if` block entirely. The subsequent flag validation (line 1204: `if (flags & ~AT_SYMLINK_NOFOLLOW != 0)`) already handles invalid flags. AT_SYMLINK_NOFOLLOW is a valid flag that requires no special handling because the VFS already operates on literal paths without following symlinks.

Rationale: SFS stores symlinks as directory entries with their own DirEntry metadata (atime, mtime). When you call setTimestamps("/mnt/mylink", ...), the VFS resolves to SFS, which finds the directory entry for "mylink" and updates its timestamps. It does not follow the symlink target. This is exactly the behavior AT_SYMLINK_NOFOLLOW requests.
