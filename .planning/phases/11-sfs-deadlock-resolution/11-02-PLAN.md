SFS currently does not implement rename (`.rename = null` in root.zig). The VFS `rename` function returns `NotSupported` when the filesystem has no rename function. This causes the rename and renameat tests to skip.

**In `src/fs/sfs/ops.zig`, add `sfsRename`:**

```zig
pub fn sfsRename(ctx: ?*anyopaque, old_path: []const u8, new_path: []const u8) vfs.Error!void {
```

Implementation approach (follows the same patterns as sfsUnlink and sfsMkdir):

1. Extract old_name and new_name by stripping leading '/' from both paths
2. Validate both names using `sfs_alloc.isValidFilename` and length < 32
3. Reject if old_name equals new_name (no-op, return success)
4. PHASE 1 (unlocked): Read full directory to find old entry index and check if new name already exists
5. If old entry not found, return NotFound
6. If old entry is a directory type, still allow rename (rename works for both files and directories)
7. PHASE 2 (under alloc_lock): Re-read the specific directory block for old entry (TOCTOU prevention)
   - Validate old entry still exists and has same name
   - If new name already exists: find its entry, check it is not a directory (cannot overwrite directory with rename), mark it for deletion (set flags=0, handle pending_delete if open)
   - Copy old entry's fields to a local DirEntry, change the name to new_name
   - Write the new entry into old entry's slot (overwrite name in place)
   - Release alloc_lock
8. PHASE 3 (outside lock): Write the modified directory sector(s) to disk
9. If the old and new entries are in different directory blocks, write both blocks

Simpler approach since SFS is flat (all entries in root directory):
1. Read directory, find old_entry_idx
2. Under lock: re-read old entry's block, validate, change the name field in-place, release lock
3. Outside lock: write the block back
4. If new_name already exists and is a different entry: under lock, mark old new_name entry as deleted (flags=0), handle pending_delete/deferred_info if open, update file_count. Release lock. Write block. Free blocks outside lock if entry was not open.

The rename is essentially: find entry with old_name, change its name to new_name. If an entry with new_name already exists, remove it first (like POSIX rename semantics).

**In `src/fs/sfs/root.zig`:**
- Change `.rename` from absent/null to `.rename = sfs_ops.sfsRename` in the FileSystem initialization (around line 89, after `.rmdir`)

**VFS rename lock concern:** VFS.rename (vfs.zig:594) holds the VFS spinlock while calling rename_fn. This means sfsRename runs with interrupts disabled (VFS lock). Inside sfsRename, acquiring alloc_lock nests another spinlock. Then readSector/writeSector acquire io_lock (third nesting level). This is acceptable because:
- Lock ordering is maintained: VFS lock > alloc_lock > io_lock
- AHCI uses polled I/O (no interrupt dependency)
- The operation is bounded (single directory, max 64 entries)

However, to minimize time under VFS lock, do the initial directory scan BEFORE acquiring alloc_lock (but still under VFS lock -- unavoidable since VFS holds it for the entire call). The write I/O should still happen under VFS lock since VFS holds it for the caller.
