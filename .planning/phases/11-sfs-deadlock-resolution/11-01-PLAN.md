<task type="auto">
  <name>Task 1: Add I/O serialization lock to SFS device access</name>
  <files>src/fs/sfs/types.zig, src/fs/sfs/io.zig</files>
  <action>
The root cause of the deadlock is a data race on `device_fd.position` in `io.zig`. Multiple threads call `readSector`/`writeSector` concurrently -- some under `alloc_lock` (e.g., `freeBlock`, `allocateBlock`), some without it (e.g., `sfsRead`, `sfsWrite` data I/O, `refreshSizeFromDisk`). The unserialized save/set/restore of `device_fd.position` causes one thread to read from the wrong LBA, corrupting filesystem state.

**In `src/fs/sfs/types.zig`:**
- Add a new `io_lock: sync.Spinlock = .{}` field to the `SFS` struct, placed after `alloc_lock`
- This lock serializes all device I/O through `readSector`/`writeSector`
- Update the lock ordering comment: `alloc_lock` (2) before `io_lock` (2.5) -- `io_lock` may be acquired while `alloc_lock` is held (current code does this), but `alloc_lock` must NEVER be acquired while `io_lock` is held

**In `src/fs/sfs/io.zig`:**
- Change `readSector` signature to take `self: *t.SFS` instead of `device_fd: *fd.FileDescriptor`
- Wrap the position save/set/read/restore sequence in `self.io_lock.acquire()` / `release()`
- Change `writeSector` signature similarly to take `self: *t.SFS`
- Wrap the position save/set/write/restore sequence in `self.io_lock.acquire()` / `release()`
- Update `readSectorAsync`, `writeSectorAsync`, `readSectorsAsync`, `writeSectorsAsync` to pass `self` through (they already take `self: *t.SFS`)
- Update `readDirectoryAsync`, `writeDirectoryAsync` similarly
- Update `updateSuperblock` to take `self: *t.SFS` and use `self.io_lock` internally (it currently calls `writeSector` which will need `self`)
- Access `self.device_fd` inside each function instead of receiving it as parameter

**Important:** The `readSector`/`writeSector` functions are called from many places in `alloc.zig` and `ops.zig`. All call sites must be updated to pass `self` (the SFS instance) instead of `self.device_fd`. This is a mechanical but widespread change. In `ops.zig`, the SFS instance is available via `file.fs` (for file operations) or `self` (for VFS operations that receive ctx). In `alloc.zig`, `self` is already the SFS instance.
  </action>
  <verify>
Run `zig build -Darch=x86_64` and `zig build -Darch=aarch64` to confirm compilation succeeds. All `readSector`/`writeSector` calls should use the new signature with `*t.SFS` instead of `*fd.FileDescriptor`.
  </verify>
  <done>
All SFS device I/O is serialized through `io_lock`. No code path accesses `device_fd.position` without holding `io_lock`. Both architectures compile successfully.

<task type="auto">
  <name>Task 2: Restructure alloc_lock usage to avoid I/O under lock</name>
  <files>src/fs/sfs/alloc.zig, src/fs/sfs/ops.zig</files>
  <action>
Currently, `freeBlock` and `allocateBlock` hold `alloc_lock` (a spinlock that disables interrupts) while performing disk I/O. Even though AHCI uses polled I/O, holding a spinlock for 3+ disk operations per `freeBlock` call (read bitmap sector, write bitmap sector, write superblock) causes extended interrupt-disabled periods. With `freeBlocks` calling `freeBlock` in a loop (potentially dozens of times for large files), this accumulates into significant interrupt starvation.

**In `src/fs/sfs/alloc.zig` -- restructure `freeBlock`:**

Replace the current single-lock-hold pattern with a two-phase approach:
1. Acquire `alloc_lock`, compute the bitmap byte index, bit index, LBA, and capture the superblock free_blocks value. Release `alloc_lock`.
2. Read the bitmap sector (now outside lock, using `io_lock` via `readSector`).
3. Modify the bitmap byte in the local buffer.
4. Write the bitmap sector back (outside `alloc_lock`).
5. Re-acquire `alloc_lock`, update `superblock.free_blocks += 1`, update bitmap cache if valid. Release `alloc_lock`.
6. Call `updateSuperblock` (outside `alloc_lock`).

The key insight: bitmap bit clearing is idempotent -- if another thread races and also clears the same bit, the result is the same. And `superblock.free_blocks` is just a counter that gets incremented.

If the bitmap cache is valid, also update it under `alloc_lock` in step 5.

**In `src/fs/sfs/alloc.zig` -- restructure `allocateBlock`:**

The current `allocateBlock` holds `alloc_lock` for the entire operation including bitmap I/O. Restructure:
1. Acquire `alloc_lock`, check bitmap cache validity. If cache is valid, scan cache for free bit, mark it, release lock, then write bitmap sector and superblock outside lock. If cache not valid, release lock.
2. If cache was not valid: load bitmap (outside lock), re-acquire lock, re-validate, scan, mark, release lock, write.

This is more complex because allocation must be atomic (two threads must not allocate the same block). The bitmap cache makes this manageable -- under `alloc_lock`, mark the bit in the cache (in-memory), release lock, then persist to disk. If persist fails, re-acquire lock and undo the cache change.

For simplicity: keep `allocateBlock` mostly as-is for now BUT move the `writeSector` and `updateSuperblock` calls outside the lock. The bitmap scan and in-memory bit-set happen under lock (fast, no I/O). The disk persistence happens after lock release. On persistence failure, re-acquire lock and undo.

Specifically:
1. Acquire `alloc_lock`
2. Load/validate bitmap cache (if cache miss, load bitmap INTO cache under lock -- this requires I/O under lock, but only on first call or cache invalidation; this is acceptable as a cold path)
3. Scan for free bit, set it in cache
4. Update superblock counters in memory
5. Capture the LBA and sector data to write
6. Release `alloc_lock`
7. Write bitmap sector to disk (outside lock)
8. Write superblock to disk (outside lock)
9. On failure of step 7 or 8: re-acquire lock, undo cache bit and counters, release, return error

**In `src/fs/sfs/ops.zig` -- fix operations that do I/O under `alloc_lock`:**

The following functions acquire `alloc_lock` then call `readSector`/`writeSector`:
- `sfsWrite` (lines 319-336): Acquires alloc_lock for directory size update, does readSector+writeSector under lock. Fix: Read directory sector BEFORE acquiring lock. Under lock, validate entry still matches (TOCTOU), update size field in buffer. Release lock. Write sector outside lock.
- `sfsChmod` (lines 933-961): Same pattern. Fix: Move writeSector call after lock release.
- `sfsChown` (lines 999-1027): Same pattern. Fix: Move writeSector call after lock release.
- `sfsMkdir` (lines 1115-1173): Acquires lock, reads sector, writes sector, updates superblock under lock. Fix: Similar two-phase approach.
- `sfsRmdir` (lines 1238-1294): Same. Fix: Similar.
- `sfsOpen` O_TRUNC path (lines 513-537): readSector+writeSector under alloc_lock. Fix: Similar.
- `sfsOpen` O_CREAT path (lines 641-696): readSector+writeSector under alloc_lock. Fix: Similar.

For ALL of these, the pattern is:
1. Read the directory sector BEFORE acquiring `alloc_lock` (or use the pre-read buffer)
2. Acquire `alloc_lock`
3. Re-read the sector under lock for TOCTOU prevention (this still does I/O under lock -- but see note below)
4. Validate, modify in-memory buffer
5. Capture the sector data and LBA
6. Release `alloc_lock`
7. Write sector to disk outside lock
8. If superblock update needed, call `updateSuperblock` outside lock

**Important note on TOCTOU re-reads:** Many operations currently re-read the directory sector under `alloc_lock` for TOCTOU prevention. This requires I/O under `alloc_lock` + `io_lock`. Since `io_lock` is a separate spinlock, this nests two spinlocks. The lock ordering must be: `alloc_lock` first, `io_lock` second. This is already the natural order since `readSector` acquires `io_lock` internally.

The TOCTOU re-reads are a correctness requirement. To eliminate I/O under `alloc_lock` entirely, we would need to validate using in-memory state only (e.g., the bitmap cache). For this phase, we accept that TOCTOU re-reads do I/O under `alloc_lock` because they are single-sector reads (fast) and are essential for correctness. The major improvement is removing the WRITE I/O and superblock updates from under the lock.

Revised approach for ops.zig: For each function, move the `writeSector` and `updateSuperblock` calls AFTER releasing `alloc_lock`. Keep `readSector` calls under lock where needed for TOCTOU. This eliminates the most expensive I/O (writes) from the locked section while preserving correctness.
