<objective>
Replace yield-loop blocking in timerfd and signalfd with proper wait queues.

Purpose: The current timerfd and signalfd implementations use `sched.yield()` in a busy loop when a blocking read has no data. This wastes CPU cycles -- the thread is repeatedly scheduled, checks the condition, and yields again. Wait queues allow the thread to sleep until woken by the relevant event (timer expiry or signal delivery), eliminating CPU spinning.

Output: Modified timerfd.zig and signalfd.zig using `sched.waitOn` / `sched.waitOnWithTimeout` for proper blocking, matching the pattern already used by eventfd.zig and futex.zig.
</objective>

<task type="auto">
  <name>Task 1: Convert timerfd blocking read from yield-loop to WaitQueue</name>
  <files>src/kernel/sys/syscall/io/timerfd.zig</files>
  <action>
Replace the yield-loop blocking in `timerfdRead` with proper wait queue sleep using the same pattern as eventfd.zig.

Current problematic pattern (lines 190-205):
```
// Block using yield loop (similar to epoll_wait pattern)
if (!state.armed) {
    held.release();
    return Errno.EAGAIN.toReturn();
}
held.release();
sched.yield();
// Retry
```

Replace with WaitQueue-based blocking:

1. Add a `wait_queue: sched.WaitQueue` field to `TimerFdState` (alongside the existing `blocked_readers` field). Initialize to `.{}` in `init()`. The `blocked_readers: ?*sched.Thread` and `reader_woken: std.atomic.Value(bool)` fields can be removed since the WaitQueue handles this.

2. In `timerfdRead`, when the timer is armed but has no expirations yet and the fd is NOT O_NONBLOCK:
   - Calculate timeout_ticks from (next_expiry_ns - now_ns), converting from nanoseconds to ticks (1 tick = ~1ms, so divide by 1_000_000 and round up).
   - Use `sched.waitOnWithTimeout(&state.wait_queue, held, timeout_ticks, null)` to sleep. This atomically releases the spinlock and blocks the thread. Pass null for futex_bucket_ptr since this is not a futex.
   - After wakeup, the while(true) loop will re-acquire the lock and re-check the condition.
   - If timer is NOT armed, return EAGAIN (unarmed timer with no expirations should not block forever).

3. In `timerfdClose`, wake all threads on the wait queue:
   - Under the lock, after setting closed=true, call `state.wait_queue.wakeUp(std.math.maxInt(usize))` to wake all blocked readers.

4. In `sys_timerfd_settime`, when rearming the timer (lines 443-448), wake blocked readers:
   - Under the lock, after updating timer state, call `state.wait_queue.wakeUp(1)` to wake one blocked reader (timer settings changed, reader should re-check).

5. Keep the `ref()` / `unref()` lifecycle pattern unchanged -- it correctly prevents use-after-free.

6. Import `sched` is already available. `sched.WaitQueue` is exported from `sched/root.zig`.

IMPORTANT: The `sched.waitOnWithTimeout` function requires that the spinlock held be passed as the second argument. It releases the lock AFTER setting the thread to Blocked state, preventing lost wakeups. Do NOT release the lock manually before calling waitOnWithTimeout.

IMPORTANT: `WaitQueue.wakeUp()` must be called while holding the state spinlock but NOT holding the scheduler lock. This is the correct lock ordering (state.lock -> scheduler.lock internally via wakeUp).

<task type="auto">
  <name>Task 2: Convert signalfd blocking read from yield-loop to WaitQueue</name>
  <files>src/kernel/sys/syscall/io/signalfd.zig</files>
  <action>
Replace the yield-loop blocking in `signalfdRead` with proper wait queue sleep.

Current problematic pattern (lines 128-136):
```
// Block - use yield loop pattern
held.release();
sched.yield();
// Retry loop - will re-check pending_signals
```

Replace with WaitQueue-based blocking:

1. Add a `wait_queue: sched.WaitQueue` field to `SignalFdState`. Initialize to `.{}` in `init()`. Remove `blocked_readers: ?*sched.Thread` and `reader_woken: std.atomic.Value(bool)` since the WaitQueue replaces them.

2. In `signalfdRead`, when no signals are pending and fd is NOT O_NONBLOCK:
   - Use `sched.waitOn(&state.wait_queue, held)` to sleep (no timeout -- signal arrival is the only wakeup condition). This atomically releases the spinlock and blocks the thread.
   - After wakeup, the while(true) loop will re-acquire the lock and re-check pending_signals.

3. In `signalfdClose`, wake all threads on the wait queue:
   - Under the lock, after setting closed=true, call `state.wait_queue.wakeUp(std.math.maxInt(usize))` to wake all blocked readers.

4. Signal delivery wakeup: When a signal is delivered to a thread (in the signal delivery path), the signalfd reader should be woken. However, the current architecture does not have a callback from signal delivery to signalfd. The signalfd state is per-FD, not per-thread, and there is no global registry of signalfd watchers.

   For now, keep the blocking behavior correct using a TIMEOUT-based approach instead:
   - Use `sched.waitOnWithTimeout(&state.wait_queue, held, 10, null)` to sleep for ~10ms at a time (matching the previous yield granularity).
   - This is still better than yield-loop because the thread is truly sleeping for 10ms rather than being scheduled, checking, and yielding every scheduler tick.
   - The TODO for proper signal-to-signalfd wakeup integration can be deferred (requires a registry of signalfd watchers that signal delivery consults).

   Alternative (better but more complex): Use `sched.waitOn` (no timeout) and add a wakeup call in `sys_kill` / `sys_rt_sigqueueinfo` when the target thread has a signalfd watching the signal. This requires checking all FDs of the target process for signalfd file ops with matching masks. Skip this for now -- the 10ms polling approach is acceptable and matches the previous behavior quality.

5. In `sys_signalfd4`, when updating mask (line 235), wake blocked readers:
   - Under the lock, after updating sigmask, call `state.wait_queue.wakeUp(1)` to wake one reader (mask changed, should re-check).

IMPORTANT: Same lock ordering rules as timerfd -- hold state.lock when calling wakeUp, do NOT hold scheduler lock.
