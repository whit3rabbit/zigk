**Part A: build.zig**

In `build.zig`, after the `create_disk_img` definition (around line 2942) and before the `run_cmd.step.dependOn` block (around line 2944), add a new build step that creates a 64MB ext2 disk image:

1. Define `const ext2_img_mb: u32 = 64;`

2. Create a shell script via `b.fmt()` that:
   - Checks for `ext2.img.stamp` sentinel file and skips if present (idempotent rebuilds)
   - Searches for `mke2fs` in order: `/opt/homebrew/opt/e2fsprogs/sbin/mke2fs`, `/usr/local/opt/e2fsprogs/sbin/mke2fs`, then `which mke2fs` (system PATH). Homebrew paths first to avoid Android SDK mke2fs getting picked up on macOS.
   - Fails with a clear error message ("ERROR: mke2fs not found. On macOS: brew install e2fsprogs") if not found.
   - Runs `dd if=/dev/zero of=ext2.img bs=1M count=64` to create the raw image.
   - Runs `"$MKE2FS" -t ext2 -b 4096 -L "zk-ext2" -m 0 ext2.img` to format it. Flags: `-t ext2` (not ext3/ext4, no journal), `-b 4096` (4KB blocks per project decision), `-L "zk-ext2"` (volume label), `-m 0` (no reserved blocks).
   - Touches `ext2.img.stamp` on success.
   - Prints success message.

3. Create the step: `const create_ext2_cmd = b.addSystemCommand(&.{ "sh", "-c", ext2_script });`

4. Wire the step into the build: add `create_ext2_cmd` dependency to `run_cmd.step` alongside the existing `create_disk_img` dependency. Place the dependOn call right after the existing `run_cmd.step.dependOn(&create_disk_img.step)` on line 2947 (inside the else branch of the `if (run_iso)` check). Also add it for the ISO path (line 2945's else block) since ext2.img is needed regardless.

IMPORTANT: The `dependOn` call must be OUTSIDE the `if (run_iso)` conditional -- ext2.img is needed for both ISO and non-ISO boot modes. Add it after line 2948 (after the closing brace of the if/else):
```
run_cmd.step.dependOn(&create_ext2_cmd.step);
```

Do NOT use `b.fmt()` with positional arguments for the script if it contains literal curly braces -- use string concatenation or escape braces as `{{` and `}}` per Zig fmt syntax. The `\\ ` multiline string syntax does not escape braces.

  <name>Task 2: Add QEMU VirtIO-SCSI ext2 device attachment for both architectures</name>
  <files>build.zig</files>
  <action>
In `build.zig`, in the QEMU run_cmd setup section, add VirtIO-SCSI attachment for ext2.img on BOTH architectures.

**aarch64** (around line 2807-2813, after the existing SFS VirtIO-SCSI block):
The aarch64 SFS block already adds `virtio-scsi-pci,id=scsi0` and `scsi-hd,drive=sfsdisk,bus=scsi0.0`. Add the ext2 disk to the SAME scsi0 controller (SCSI supports multiple LUNs). Add these args inside the existing `if (target_arch == .aarch64)` block, right after the SFS device line (line 2811):
```zig
run_cmd.addArgs(&.{
    "-drive", "file=ext2.img,format=raw,if=none,id=ext2disk",
    "-device", "scsi-hd,drive=ext2disk,bus=scsi0.0",
});
```

**x86_64** (around line 2813, after the aarch64 SFS block ends):
x86_64 currently does NOT have a VirtIO-SCSI controller. Add one specifically for ext2. Create a new block after the aarch64 SFS block:
```zig
// ext2 storage disk (both architectures use VirtIO-SCSI for consistency)
if (target_arch == .x86_64) {
    run_cmd.addArgs(&.{
        "-device", "virtio-scsi-pci,id=scsi0",
        "-drive", "file=ext2.img,format=raw,if=none,id=ext2disk",
        "-device", "scsi-hd,drive=ext2disk,bus=scsi0.0",
    });
}
```

This means:
- aarch64: scsi0 controller already exists (SFS), ext2 is a second LUN on it
- x86_64: scsi0 controller is new (only for ext2), SFS continues using AHCI on boot disk

The existing x86_64 SFS path uses AHCI on the boot disk (disk.img), which is unaffected by adding VirtIO-SCSI.

CRITICAL: For aarch64, do NOT add a second `virtio-scsi-pci` controller. The ext2 drive should be on the existing `scsi0` bus that was created for SFS. SCSI allows multiple devices on one controller.
