  <files>src/kernel/sys/syscall/process/process.zig</files>
  <action>
Modify sys_setregid (line 523) to enforce POSIX permission checks without hasSetGidCapability bypass.

Current bug: Lines 538 and 545 check `!proc.hasSetGidCapability(new_rgid)` which returns true for ANY group in the supplementary groups list. After setresgid(1000,1000,1000), a process with groups [1000, 2000] can call setregid(2000, 2000) because hasSetGidCapability(2000) returns true.

POSIX requirement: Unprivileged process can only set rgid/egid to:
- Current real GID (proc.gid)
- Current effective GID (proc.egid)
- Saved set-GID (proc.sgid)

Fix: Remove hasSetGidCapability check entirely. Use canSetGid helper pattern from sys_setresgid (line 412-417):

```zig
fn canSetGid(proc: *Process, target_gid: u32) bool {
    return target_gid == proc.gid or target_gid == proc.egid or target_gid == proc.sgid;
}
```

Update permission checks:
- Line 538-542: `if (!is_privileged and !canSetGid(proc, new_rgid)) return error.EPERM;`
- Line 545-549: `if (!is_privileged and !canSetGid(proc, new_egid)) return error.EPERM;`

Simplify nested conditions to single-line checks. Remove hasSetGidCapability calls.

This matches the fix pattern already used in sys_setresgid (process.zig:412-417).

  <name>Task 2: Fix copyStringFromUser stack buffer validation (BUGFIX-03)</name>
  <files>src/kernel/sys/syscall/core/user_mem.zig</files>
  <action>
Fix copyStringFromUser (line 54) to accept stack-allocated userspace buffers.

Current bug: Line 60 calls `isValidUserAccess(src, 1, .Read)` which performs page mapping verification. The comment at lines 123-137 documents that page presence checks are SKIPPED for demand paging (stack buffers declared but not yet faulted-in). However, the isValidUserAccess call at line 60 happens BEFORE the assembly copy, defeating the fixup mechanism.

Root cause: copyStringFromUser validates the pointer before calling _asm_copy_from_user. For stack buffers not yet faulted-in, isValidUserAccess returns false because the page isn't present, even though the assembly copy would handle the fault correctly.

Fix: Remove the isValidUserAccess check at line 60. Replace with fast bounds check only:

```zig
pub fn copyStringFromUser(dest: []u8, src: usize) ![]u8 {
    if (dest.len == 0) return &[_]u8{};
    if (src == 0) return error.Fault;

    // Fast bounds check only - assembly copy handles page faults via fixup
    if (!isValidUserPtr(src, 1)) return error.Fault;

    // Use raw copy to pull as much as possible
    const rem = _asm_copy_from_user(dest.ptr, @ptrFromInt(src), dest.len);
    // ... rest unchanged
}
```

The assembly _asm_copy_from_user uses the fixup mechanism (documented in CLAUDE.md) to handle page faults gracefully. If a page isn't accessible, the copy stops and returns rem > 0, which triggers error.Fault at line 74.

This pattern matches copyFromUser (line 149) which also uses only isValidUserPtr for bounds checking, then relies on _asm_copy_from_user fixup for page fault handling.

Implementation:
1. Add sfsChown function following existing SFS operation patterns (sfsRead, sfsWrite, etc.):

```zig
pub fn sfsChown(file_desc: *fd.FileDescriptor, uid: u32, gid: u32) isize {
    const held = file_desc.lock.acquire();
    defer held.release();

    const file: *t.SfsFile = @ptrCast(@alignCast(file_desc.private_data));

    // Update in-memory metadata
    file.uid = uid;
    file.gid = gid;

    // Write metadata to disk
    const inode_block = file.fs.superblock.inode_table_start + (file.inode_num / (512 / @sizeOf(t.SfsInode)));
    const inode_offset = (file.inode_num % (512 / @sizeOf(t.SfsInode))) * @sizeOf(t.SfsInode);

    var sector: [512]u8 align(512) = undefined;
    sfs_io.readSectorsSync(file.fs, inode_block, 1, &sector) catch return -5; // EIO

    const inode: *t.SfsInode = @ptrCast(@alignCast(&sector[inode_offset]));
    inode.uid = uid;
    inode.gid = gid;

    sfs_io.writeSectorsSync(file.fs, inode_block, 1, &sector) catch return -5; // EIO

    return 0;
}
```

2. Update SFS file_ops struct to reference sfsChown:

```zig
pub const sfs_file_ops = fd.FileOps{

    .write = sfsWrite,
    .poll = sfsPoll,
    .chown = sfsChown,  // Add this line
    // ... rest unchanged
};
```

Pattern follows existing SFS operations: lock FD, cast private_data to SfsFile, read inode sector, modify in-place, write back. Uses synchronous I/O like other SFS metadata operations.
