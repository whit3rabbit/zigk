<task type="auto">
  <name>Task 1: Acquire tcb.mutex before transmitPendingData in TCP_CORK uncork</name>
  <files>src/net/transport/socket/options.zig</files>
  <action>
In `setsockopt()`, the TCP_CORK handling (around lines 163-178) calls `transmitPendingData(tcb)` when cork is cleared, but only holds `sock.lock` (acquired at line 18). It does NOT hold `tcb.mutex`, which violates the locking pattern used by all other TCB mutation paths (see tcp/api.zig send/recv/close all acquire tcb.mutex).

The existing comment at line 172 says "transmitPendingData does not acquire tcb.mutex -- it is called from the owning thread context here." This comment is wrong/misleading -- the issue is not about who calls it but about concurrent access from the TCP RX path or timer path, both of which also call transmitPendingData under tcb.mutex.

**Fix:** In the TCP_CORK arm, when `!new_cork`, acquire `tcb.mutex` before calling `transmitPendingData()` and release after. The lock ordering is safe: sock.lock (level 6) is already held, and tcb.mutex (level 7) comes after it in the lock ordering hierarchy.

Replace the block:
```zig
                    // When cork is cleared, flush pending data.
                    // We are under sock.lock. transmitPendingData does not acquire
                    // tcb.mutex -- it is called from the owning thread context here.
                    if (!new_cork) {
                        const tx_data = @import("../tcp/tx/data.zig");
                        _ = tx_data.transmitPendingData(tcb);
                    }
```

With:
```zig
                    // When cork is cleared, flush pending data.
                    // Acquire tcb.mutex to match locking pattern of all other
                    // TCB mutation paths (RX processing, timer retransmit, send).
                    // Lock order: sock.lock (level 6) -> tcb.mutex (level 7).
                    if (!new_cork) {
                        const tx_data = @import("../tcp/tx/data.zig");
                        const tcb_held = tcb.mutex.acquire();
                        defer tcb_held.release();
                        _ = tx_data.transmitPendingData(tcb);
                    }
```
  </action>
  <verify>
`zig build -Darch=x86_64` compiles without errors. Grep `options.zig` for `mutex.acquire` and confirm it appears inside the TCP_CORK uncork path immediately before `transmitPendingData`.
  </verify>
  <done>
TCP_CORK uncork flush in setsockopt acquires tcb.mutex before calling transmitPendingData(), matching the locking pattern used in all other TCB mutation paths (RX, timers, send).
  </done>

<task type="auto">
  <name>Task 2: Implement MSG_DONTWAIT and MSG_PEEK for raw socket recv</name>
  <files>src/net/transport/socket/raw_api.zig</files>
  <action>
Both `recvfromRaw()` and `recvfromRaw6()` currently discard the flags parameter (`_ = flags;`). They need to handle MSG_DONTWAIT and MSG_PEEK the same way TCP recv does.

**MSG_DONTWAIT behavior:** If set, override `sock.blocking` -- treat the call as non-blocking regardless of the socket's blocking mode. If no data is available, return WouldBlock immediately.

**MSG_PEEK behavior:** If set, use `sock.peekPacketIp()` instead of `sock.dequeuePacketIp()` to read data without consuming it from the queue.

**Fix `recvfromRaw()` (IPv4 raw recv):**

1. Remove the `_ = flags;` line (line 129).

2. Determine effective blocking mode: `const is_nonblocking = (flags & types.MSG_DONTWAIT) != 0 or !sock.blocking;`

3. Determine if peek: `const is_peek = (flags & types.MSG_PEEK) != 0;`

4. In the data dequeue section (inside `sock.lock` held region, around line 147), use `peekPacketIp` when `is_peek` is true, otherwise use `dequeuePacketIp`:
```zig
const maybe_len = if (is_peek)
    sock.peekPacketIp(buf, &ip_addr, &src_port)
else
    sock.dequeuePacketIp(buf, &ip_addr, &src_port);

if (maybe_len) |len| {
    // ... existing success path ...
}
```

5. In the blocking check (around line 164), use `is_nonblocking` instead of `!sock.blocking`:
```zig
if (is_nonblocking) {
    held.release();
    state.releaseSocket(sock);
    return errors.SocketError.WouldBlock;
}
```

**Fix `recvfromRaw6()` (IPv6 raw recv):**

Apply the exact same changes as above to `recvfromRaw6()`:
1. Remove `_ = flags;` (line 305)
2. Add `is_nonblocking` and `is_peek` flags
3. Use `peekPacketIp` vs `dequeuePacketIp` based on `is_peek`
4. Use `is_nonblocking` in the blocking check

The blocking loop itself (lines 172-188 / 349-365) does not need to change -- it already correctly sets blocked_thread and calls block_fn(). The only change is the condition that decides whether to enter the blocking loop.
