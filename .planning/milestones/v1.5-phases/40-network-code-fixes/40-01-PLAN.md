<task type="auto">
  <name>Task 1: Clear tcb.blocked_thread before EINTR in both TCP recv paths</name>
  <files>src/kernel/sys/syscall/net/net.zig</files>
  <action>
In `sys_recvfrom` (around line 595), there are two blocking recv paths that set `tcb.blocked_thread = current` before calling `sched.block()`, then only clear `sock_for_type.blocked_thread = null` but forget to clear `tcb.blocked_thread` before returning EINTR.

**Fix 1 -- MSG_PEEK blocking path (around lines 619-627):**
After `sched.block()` returns and before the `hasPendingSignal()` check, add `tcb.blocked_thread = null` clearing. The block at lines 619-621 sets `tcb.blocked_thread = current` under a `getTcb()` check. After `sched.block()` returns (line 623-624), `sock_for_type.blocked_thread` is cleared but `tcb.blocked_thread` is not.

Change the code so that after `sched.block()`, we also clear `tcb.blocked_thread`:

```zig
sched.block();
sock_for_type.blocked_thread = null;
// Clear tcb.blocked_thread to prevent stale pointer on retry
if (socket.getTcb(ctx.socket_idx)) |tcb| {
    tcb.blocked_thread = null;
}
if (hasPendingSignal()) {
    return error.EINTR;
}
```

**Fix 2 -- Default blocking TCP recv path (around lines 662-670):**
Same pattern. After `sched.block()` (line 666), `sock_for_type.blocked_thread` is cleared (line 667) but `tcb.blocked_thread` is not.

Change the code so that after `sched.block()`, we also clear `tcb.blocked_thread`:

```zig
sched.block();
sock_for_type.blocked_thread = null;
// Clear tcb.blocked_thread to prevent stale pointer on retry
if (socket.getTcb(ctx.socket_idx)) |tcb| {
    tcb.blocked_thread = null;
}
if (hasPendingSignal()) {
    return error.EINTR;
}
```

IMPORTANT: Use `socket.getTcb(ctx.socket_idx)` to re-fetch the TCB safely (it does acquire/release internally), since the TCB might have been freed while we were blocked. Do NOT use a stale TCB pointer captured before blocking.
  </action>
  <verify>
`zig build -Darch=x86_64` completes without errors. Grep for `tcb.blocked_thread = null` in `net.zig` and confirm it appears in both the MSG_PEEK blocking path and the default blocking recv path, each occurring after `sched.block()` and before the `hasPendingSignal()` check.
  </verify>
  <done>
Both MSG_PEEK blocking and default TCP blocking recv paths in net.zig clear tcb.blocked_thread to null after wakeup, before checking for pending signals and potentially returning EINTR. No stale tcb.blocked_thread pointer survives a signal-interrupted recv retry.
  </done>
</task>

<task type="auto">
  <name>Task 2: Propagate SO_RCVBUF and SO_SNDBUF from socket to TCB in connect paths</name>
  <files>src/net/transport/socket/tcp_api.zig</files>
  <action>
In `tcp_api.zig`, the `connect()` function (line 209) copies `nodelay` and `tos` from sock to tcb after `connectIp()` returns, but does NOT copy `rcv_buf_size` or `snd_buf_size`. This means buffer sizes set via setsockopt before connect() are lost.

**Fix 1 -- `connect()` (around line 217):**
After the existing `tcb.tos = sock.tos;` line, add:
```zig
tcb.rcv_buf_size = sock.rcv_buf_size;
tcb.snd_buf_size = sock.snd_buf_size;
```

**Fix 2 -- `connect6()` (around line 330):**
After the existing `tcb.tos = sock.tos;` line, add the same two lines:
```zig
tcb.rcv_buf_size = sock.rcv_buf_size;
tcb.snd_buf_size = sock.snd_buf_size;
```

**Fix 3 -- `connectAsync()` (around line 847):**
After the existing `tcb.tos = sock.tos;` line, add the same two lines.

**Fix 4 -- `connectAsync6()` (around line 914):**
After the existing `tcb.tos = sock.tos;` line, add the same two lines.

These four functions are the only paths where a new TCB is created for outgoing connections. The `listen()` path does not need this fix because accepted connections inherit buffer sizes from the listening TCB, not the socket directly.
  </action>
  <verify>
`zig build -Darch=x86_64` completes without errors. Grep for `rcv_buf_size` in `tcp_api.zig` and confirm it appears in all four connect functions: `connect`, `connect6`, `connectAsync`, `connectAsync6`.
  </verify>
  <done>
All four connect paths (connect, connect6, connectAsync, connectAsync6) copy rcv_buf_size and snd_buf_size from the socket to the newly created TCB, ensuring SO_RCVBUF/SO_SNDBUF values set before connect() take effect on the TCP connection.
  </done>
