<objective>
Initialize the kernel's loopback network interface at boot and wire it into the full network stack so that TCP/UDP connections to 127.0.0.1 work on both x86_64 and aarch64.

Purpose: The loopback driver (`src/net/drivers/loopback.zig`) exists but is never initialized. The `initNetwork()` function currently calls `initSyscallOnly()` which sets the global interface to null, causing all TCP/UDP operations to return NetworkDown. Initializing the loopback interface and the full network stack enables guest-internal networking without any QEMU configuration changes (loopback is purely in-kernel software).

Output: Modified `src/kernel/core/init_hw.zig` with loopback init, full stack init, and combined tick callback.
</objective>

<task type="auto">
  <name>Task 2: Wire net.tick() into scheduler tick callback for TCP timers</name>
  <files>src/kernel/core/init_hw.zig</files>
  <action>
The scheduler has a single `tick_callback` slot. On aarch64, this is already set to `usbPollTickCallback` for USB event polling. TCP timers (retransmission, keepalive, connection timeout) require `net.tick()` to be called periodically.

1. **Replace `usbPollTickCallback` with a combined callback** that does both USB polling and network ticking. Rename or create a new function `combinedTickCallback`:
   ```zig
   fn combinedTickCallback() void {
       // Network stack timers (TCP retransmission, ARP expiry, mDNS)
       net.tick();

       // USB event polling (aarch64 only -- no MSI-X, requires polling)
       if (builtin.cpu.arch == .aarch64) {
           _ = usb.xhci.pollEvents();
       }
       usb_poll_counter +%= 1;
   }
   ```

2. **Set `combinedTickCallback` unconditionally** after initNetwork (regardless of architecture). Currently the tick callback is only set on aarch64 when USB polling is needed. Move the `sched.setTickCallback(combinedTickCallback)` call to the END of `initNetwork()`, BEFORE the function returns. This ensures net.tick() runs on both x86_64 and aarch64:
   ```zig
   // Register combined tick callback for network timers + USB polling
   sched.setTickCallback(combinedTickCallback);
   console.debug("[NETSTACK] Registered tick callback for network timers", .{});
   ```

3. **Remove the old `sched.setTickCallback(usbPollTickCallback)` call** from `initUsb()` (around line 481). The combined callback now handles USB polling too. Keep the `usbPollTickCallback` function or remove it -- it's no longer needed.

4. **Guard the USB polling in the combined callback**: The combined callback should only call `usb.xhci.pollEvents()` if (a) we're on aarch64, AND (b) a controller exists without MSI-X. Use a simple flag or check:
   ```zig
   fn combinedTickCallback() void {
       net.tick();
       // USB polling only needed on aarch64 where MSI-X is unavailable
       if (comptime builtin.cpu.arch == .aarch64) {
           if (usb.xhci.getController()) |ctrl| {
               if (ctrl.msix_vectors == null) {
                   _ = usb.xhci.pollEvents();
               }
           }
       }
   }
   ```

5. **Ensure `sched` import exists** in init_hw.zig. Check that `const sched = @import("sched")` is already present (it should be, since `setTickCallback` was already called from `initUsb`).

NOTE: The tick callback runs in interrupt context (timer ISR). `net.tick()` calls `arp.tick()`, `tcp.tick()`, `mdns.tick()`, and `io.timerTick()`. These are all designed to be fast (increment counters, check timeouts) and should be safe in ISR context. The io.timerTick is already called from this path.

Grep for `usbPollTickCallback` -- it should either be removed or unreferenced. Grep for `net.tick()` in init_hw.zig to confirm it's called from the combined callback.
