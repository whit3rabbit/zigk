**Step 3: Add integration test** (seccomp.zig):
Add a test `testSeccompFilterInstructionPointer` that verifies the instruction_pointer field is populated:
1. Set PR_SET_NO_NEW_PRIVS via prctl (required for seccomp)
2. Build a BPF program that:
   - Loads instruction_pointer at offset 8 (offsetof SeccompData.instruction_pointer, low 32 bits)
   - Checks if it's non-zero (JEQ 0 -> KILL, otherwise ALLOW)
   - This ensures the kernel is actually populating the field
3. Install the filter via seccomp(SECCOMP_SET_MODE_FILTER, ...)
4. Make a syscall (e.g., getpid) -- if instruction_pointer is populated (non-zero), the filter allows it
5. If getpid succeeds, the test passes (instruction_pointer was non-zero)
6. If getpid fails with ENOSYS (killed), the instruction_pointer was still 0 and the fix didn't work

BPF program structure:
```
BPF_LD | BPF_W | BPF_ABS, k=8    // Load instruction_pointer low 32 bits
BPF_JMP | BPF_JEQ | BPF_K, k=0, jt=1, jf=0  // If zero, jump to kill
BPF_RET | BPF_K, k=SECCOMP_RET_ALLOW   // Non-zero: allow
BPF_RET | BPF_K, k=SECCOMP_RET_KILL_THREAD  // Zero: kill (test fails)
```

Wait -- instruction_pointer is u64 at offset 8 in SeccompData. On both architectures, user-space code runs at addresses that have non-zero low 32 bits (addresses like 0x200000-0x400000 for user code). Loading the low 32 bits at offset 8 and checking non-zero is sufficient.

Actually, be careful: offset 8 loads bytes 8-11 of SeccompData. SeccompData layout is: nr(4 bytes, offset 0) + arch(4 bytes, offset 4) + instruction_pointer(8 bytes, offset 8) + args(48 bytes, offset 16). So offset 8 loads the LOW 32 bits of instruction_pointer on little-endian. This is what we want.

This test must run in a child process (fork) since seccomp is permanent. Use the same runInChild pattern as existing seccomp tests.
