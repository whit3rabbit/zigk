### Integration Defect

**INT-01: Stale tcb.blocked_thread on EINTR** (low severity)

- **File:** `src/kernel/sys/syscall/net/net.zig`
- **Lines:** 619-627 (MSG_PEEK blocking), 662-670 (default TCP blocking)
- **Issue:** Both paths set `tcb.blocked_thread = current` before `sched.block()` but do not clear it before `return error.EINTR`. `sock_for_type.blocked_thread` is correctly cleared (line 624, 667), but the TCB copy is not.
- **Consequence:** If a signal interrupts a blocking recv and data subsequently arrives before the thread calls into the socket again, `established.zig` will call `socket.wakeThread(tcb.blocked_thread)` on an already-running thread.
- **Severity:** Low. Requires specific signal + data timing. Common case (data arrival wakes the thread) works correctly.
- **Fix:** Add `if (socket.getTcb(ctx.socket_idx)) |tcb| { tcb.blocked_thread = null; }` immediately before the EINTR check at lines 625 and 668.
- **Affected Requirements:** API-01 (MSG_PEEK EINTR path)

### Previously Identified Integration Concerns (from prior audit, carried forward)

**Pre-connect buffer propagation:** `setsockopt(SO_RCVBUF/SO_SNDBUF)` before `connect()` stores the value on Socket but `Tcb.init()` creates a new TCB with defaults, ignoring Socket-level settings. Minor -- most applications set options after connect. (BUF-01, BUF-02)

**TCP_CORK flush locking:** `options.zig` calls `transmitPendingData()` under `sock.lock` but without `tcb.mutex`. Concurrent `processTimers()` creates a theoretical race on `snd_nxt`/`retrans_timer`. Minor in single-core context. (API-04)
